본 문서는 AI법친의 **문서 템플릿(document template)** 이 어떤 구조를 가지며,  
질문셋 답변과 어떻게 연결되고,  
문단 단위 생성·재생성·잠금·승인·출력과 어떤 관계를 가지는지를 정의한다.

문서 템플릿은 단순한 문서 제목 목록이 아니라, 아래를 포함하는 **생성 구조체**로 본다.

- 템플릿 기본 정보  
- 섹션 구조  
- 문단 구조  
- 질문셋 답변 매핑 규칙  
- 문단 생성 방식  
- 잠금·승인·출력 정책 연결점

## 1. 목적

이 문서의 목적은 다음과 같다.

1. 문서 템플릿의 기본 구조를 고정한다.  
2. 문서 템플릿과 질문셋의 연결 기준을 정의한다.  
3. 문단 단위 생성·재생성·잠금의 기준을 정의한다.  
4. 승인 전 초안과 승인 후 산출물의 구조적 차이를 정리한다.  
5. 문서 템플릿 관리자와 실제 문서 생성 로직이 같은 기준을 따르게 한다.  
6. Phase 2의 AI 출력 정책서와 연결될 수 있는 구조를 선행 정리한다.

## 2. 문서 템플릿의 위치

문서 템플릿은 사건 라이프사이클상 아래 구간에 직접 연결된다.

```text
IN_INTERVIEW
→ INTERVIEW_DONE
→ DRAFTING
→ REVIEW_PENDING
→ APPROVED
→ DELIVERED
```

**의미**

- 질문셋 답변이 충분히 모여 `INTERVIEW_DONE` 이 되면 문서 생성 입력이 준비된다.  
- 문서 템플릿은 `DRAFTING` 단계에서 실제 초안 구조를 만든다.  
- 검토 단계(`REVIEW_PENDING`)에서 문단 단위 수정·재생성·잠금이 일어난다.  
- 승인(`APPROVED`) 이후에는 승인본 기준 출력 구조가 잠긴다.  

즉, 문서 템플릿은 질문셋 답변을 실제 법률 문서 구조로 변환하는 핵심 기준이다.

---

## 3. 문서 템플릿 구성 단위

문서 템플릿은 아래 단위로 본다.

### 3-1. 템플릿(Template)

하나의 문서 유형 전체를 정의하는 상위 단위.

**예**

- 의견서 템플릿  
- 사실관계 요약서 템플릿  
- 사건 진행 설명문 템플릿  
- 진술 정리문 템플릿  

### 3-2. 섹션(Section)

문서 안에서 문단들을 묶는 상위 구간.

**예**

- 사건 개요  
- 사실관계 정리  
- 주요 쟁점  
- 법률 검토  
- 결론 및 요청사항  

### 3-3. 문단(Paragraph)

실제 생성·재생성·잠금·승인 단위가 되는 최소 작성 블록.

### 3-4. 매핑 규칙(Mapping Rule)

질문셋 답변이 어떤 섹션·문단으로 들어갈지 정하는 규칙.

## 7. 문단 생성 방식 정의

문단은 아래 생성 방식을 기본으로 가진다.

| 값 | 설명 |
|----|------|
| ai_generate | 질문셋 답변을 바탕으로 AI가 본문 초안 생성 |
| ai_assist | 기초 문장을 만들고 담당자가 수정 중심 |
| manual_only | 자동 생성 없이 수동 입력만 허용 |
| locked_reference | 기준 문구·고지문처럼 사실상 고정 텍스트 |

**운영 원칙**

- 같은 문단 생성 방식은 관리자 화면과 실제 생성 로직에서 동일하게 해석되어야 한다.  
- `locked_reference` 문단은 재생성 대상이 아니다.  
- `manual_only` 문단은 질문셋 매핑 없이 담당자 입력 중심일 수 있다.

## 8. 질문셋 매핑 규칙

문서 템플릿은 [QUESTION_SET_DEFINITION.md](./QUESTION_SET_DEFINITION.md) 의 질문 키와 연결되어야 한다.

### 8-1. mappingSources

각 문단은 하나 이상의 질문 키를 입력으로 받을 수 있다.

**예**

- `case_overview` 문단 ← `case_title`, `fact_summary`  
- `issue_summary` 문단 ← `issue_list`  
- `timeline_description` 문단 ← `timeline`  
- `request_summary` 문단 ← `requested_outcome`  

### 8-2. 매핑 방식

| 방식 | 설명 |
|------|------|
| direct | 질문 답변을 직접 반영 |
| summary | 여러 답변을 요약 |
| merge | 복수 질문을 병합 |
| conditional | 특정 조건에서만 문단 생성 |
| attachment_linked | 첨부자료와 연결해 생성 |

### 8-3. 원칙

- 질문셋의 `mappingTargets` 와 템플릿의 `mappingSources` 는 같은 명명 규칙을 써야 한다.  
- 질문 키가 바뀌면 문단 생성 영향도를 점검해야 한다.  
- 필수 문단의 입력 키가 비어 있으면 생성 전에 경고 또는 차단 규칙이 필요하다.

## 10. 문서 생성 규칙

문서 생성은 아래 기본 조건을 따른다.

### 10-1. 생성 가능 조건

- 사건 상태가 문서 생성 가능한 구간이어야 한다.  
- 일반적으로 `INTERVIEW_DONE`, `DRAFTING`, `REVIEW_PENDING`  
- 질문셋 필수 답변이 충분해야 한다.  
- 템플릿이 활성 상태여야 한다.  
- 생성 권한이 있는 역할이어야 한다.  

### 10-2. 생성 결과

문서 생성 시 아래가 함께 준비되어야 한다.

- 문서 메타 정보  
- 섹션 구조  
- 문단 구조  
- 각 문단의 생성 초안  
- 생성 이력  
- 재생성 가능 상태  

---

## 11. 문단 재생성 규칙

문단은 전체 문서를 다시 만들지 않고 문단 단위 재생성이 가능해야 한다.

**원칙**

- 재생성은 문단 단위로 수행한다.  
- 잠긴 문단은 원칙적으로 자동 재생성 대상에서 제외한다.  
- 재생성 시 기존 문단과의 비교 근거가 남아야 한다.  
- 승인된 문단은 재생성 시 별도 확인 절차가 필요하다.  

**재생성 대상 예시**

- 사실관계 요약 문단  
- 쟁점 정리 문단  
- 법률 검토 문단  

**재생성 제외 예시**

- 승인 문구  
- 고지문·면책문구  
- 검증코드·출력 고정문구  
- 잠금된 기준 문단  

---

## 12. 문단 잠금 규칙

문단 잠금은 품질 고정과 승인 추적을 위한 기능이다.

**원칙**

- 잠긴 문단은 일반 재생성 대상에서 제외  
- 잠금 해제는 권한 있는 역할만 가능  
- 승인된 문단 잠금은 더 높은 수준의 보호가 필요  
- 잠금·해제 이력은 남아야 한다  

**잠금 대상 예시**

- 최종 승인된 결론 문단  
- 변호사 확인이 끝난 법률 검토 문단  
- 대외 전달용 설명 문단  

---

## 13. 승인본과 비승인본 구분

문서 템플릿은 승인 전·후 문서를 같은 것으로 보지 않는다.

### 13-1. 비승인본

- 초안 상태  
- 재생성 가능  
- 수정 가능  
- 워터마크·고지문 필요 가능성 있음  

### 13-2. 승인본

- 승인 기준 충족  
- 승인 시점 구조가 기준  
- 승인본 잠금 가능  
- 출력·전달의 기준이 됨  

**원칙**

- 승인 전 문서와 승인 후 문서는 정책상 다르게 취급한다.  
- 승인본 출력 규칙은 후속 [AI_OUTPUT_POLICY.md](./AI_OUTPUT_POLICY.md) 와 연결한다.

## 16. 템플릿 버전 정책

문서 템플릿은 버전 관리 대상이다.

**원칙**

- 배포 후 본문 구조를 직접 덮어쓰기보다 버전 갱신을 우선 고려한다.  
- 기존 사건은 연결 당시 템플릿 버전 기준으로 재현 가능해야 한다.  
- 템플릿 구조가 바뀌면 문단 매핑·출력 정책 영향도를 함께 검토해야 한다.  

**운영 상태 예시**

- `draft`  
- `published`  
- `archived`

## 18. 구현 대조 체크포인트

구현과 대조할 때는 아래를 본다.

- 템플릿 관리자와 실제 문서 생성 로직이 같은 구조를 쓰는가  
- 질문셋 답변과 문단 매핑이 실제로 연결되는가  
- 문단 재생성·잠금·복원이 문서 구조에 맞게 동작하는가  
- 승인본과 비승인본이 구분되는가  
- 템플릿 버전이 보존되는가  
- 템플릿 변경이 기존 사건에 미치는 영향이 통제되는가

## 19. Phase 1 정렬 관점의 의미

Phase 1 기준으로 문서 템플릿 정의서는 아래와 연결된다.

| 기준 문서 | 역할 |
|-----------|------|
| 상태값 정의서 | 문서 생성 가능한 상태 구간 정의 |
| 사건 라이프사이클 정의서 | 문서 작성·검토·승인 흐름 정의 |
| 권한정의서 | 누가 문서를 생성·승인·잠글 수 있는지 정의 |
| 질문셋 정의서 | 문서 생성 입력 구조 정의 |
| **문서 템플릿 정의서(본 문서)** | 실제 출력 구조 정의 |

즉, 본 문서는 Phase 1의 마지막 큰 축으로서 질문 입력 구조를 문서 출력 구조로 바꾸는 기준선이다.
