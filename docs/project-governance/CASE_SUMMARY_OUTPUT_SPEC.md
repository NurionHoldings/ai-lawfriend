본 문서는 AI법친에서 생성·표시되는 **사건 요약(case summary)** 의 구조, 포함 항목, 출력 규칙, 노출 대상 차이, 금지 표현, 상태별 처리 원칙을 정의한다.

사건 요약은 단순 짧은 문장이 아니라 아래를 포함하는 **구조화된 출력 산출물**로 본다.

- 사건 핵심 개요  
- 주요 사실관계  
- 쟁점 요약  
- 현재 상태  
- 다음 단계  
- 주의·누락 정보  
- 노출 대상별 차등 문구  

---

## 1. 목적

이 문서의 목적은 다음과 같다.

1. 사건 요약 출력 구조를 고정한다.  
2. 질문셋 답변을 어떤 방식으로 요약할지 기준을 만든다.  
3. 내부용과 외부용 사건 요약을 구분한다.  
4. 사건 상태에 따라 어떤 요약이 허용되는지 정의한다.  
5. 승인 전 초안과 확정 산출물의 혼동을 막는다.  
6. 후속 입력·출력 데이터 정의서와 API 명세의 기준이 된다.  

---

## 2. 사건 요약의 위치

사건 요약은 아래 흐름에서 사용된다.

```text
IN_INTERVIEW
→ INTERVIEW_DONE
→ DRAFTING
→ REVIEW_PENDING
→ APPROVED
```

**의미**

- 인터뷰 중에는 중간 요약이 가능하다.  
- 인터뷰 완료 후에는 구조화된 사건 요약 초안이 생성될 수 있다.  
- 문서 작성·검토 단계에서는 내부 검토용 사건 요약으로 사용될 수 있다.  
- 승인 이후에도 사건 개요 표시용 요약은 남을 수 있으나, 법률 의견서 자체를 대체하지는 않는다.  

---

## 3. 사건 요약의 정의

사건 요약은 사건 전체를 대신하는 최종 문서가 아니라,  
사건의 현재 상태와 핵심 내용을 빠르게 이해하기 위한 구조화 출력이다.

**사건 요약이 하는 일**

- 사건의 현재 맥락을 빠르게 전달  
- 인터뷰 내용의 핵심 정리  
- 문서 작성 전 입력 구조화  
- 담당자·관리자·의뢰인에게 필요한 수준의 설명 제공  

**사건 요약이 하지 않는 일**

- 최종 법률 판단 확정  
- 승인 문서 대체  
- 법률 의견서 자체 역할 수행  
- 근거 없는 사실 확정  

---

## 4. 출력 유형 분류

사건 요약은 아래 4종으로 나눈다.

### 4-1. 인터뷰 중간 요약

- 인터뷰 도중 표시  
- 현재까지 입력된 정보를 정리  
- 누락 정보와 추가 질문 유도 가능  

### 4-2. 인터뷰 완료 후 사건 요약 초안

- `INTERVIEW_DONE` 시점 기준  
- 문서 생성 입력용 구조화 요약  
- 내부 검토 중심  

### 4-3. 내부 검토용 사건 요약

- `DRAFTING`, `REVIEW_PENDING` 구간에서 사용  
- 담당자·관리자가 빠르게 사건을 파악하기 위한 요약  
- 쟁점, 누락, 보완 필요사항 포함 가능  

### 4-4. 의뢰인 설명용 사건 요약

- 외부·의뢰인에게 보여줄 수 있는 설명형 요약  
- 내부 검토 표현 최소화  
- 확정 의견처럼 보이지 않도록 제한  

---

## 5. 사건 요약 기본 구조

사건 요약 1개는 최소 아래 필드를 가진다.

| 필드 | 설명 |
|------|------|
| summaryId | 요약 고유 ID |
| caseId | 사건 ID |
| summaryType | 요약 유형 |
| version | 요약 버전 |
| statusSnapshot | 생성 시점 사건 상태 |
| generatedAt | 생성 시각 |
| generatedBy | 생성 주체(AI·사용자·시스템) |
| audience | 내부·외부 노출 대상 |
| sections | 요약 섹션 목록 |
| isApprovedSnapshot | 승인 기준 여부 |
| noticePolicy | 적용 고지문 정책 식별자 |

## 6. 사건 요약 섹션 구조

사건 요약은 아래 섹션을 기본으로 가진다.

| 섹션 | 설명 | 필수 여부 |
|------|------|-----------|
| `case_overview` | 사건 기본 개요 | 필수 |
| `fact_summary` | 핵심 사실관계 | 필수 |
| `issue_summary` | 주요 쟁점 | 권장 |
| `status_summary` | 현재 사건 상태 설명 | 필수 |
| `next_step` | 다음 진행 단계 | 권장 |
| `risk_or_missing_info` | 누락 정보·주의사항 | 내부용 권장 |
| `request_summary` | 의뢰인 요청사항 요약 | 권장 |

**내부용 추가 가능 섹션**

- `review_note`  
- `document_preparation_note`  
- `missing_evidence`  

**외부용 제한 원칙**

의뢰인용 사건 요약에는 아래를 원칙적으로 직접 노출하지 않는다.

- 내부 검토 메모  
- 승인 대기 내부 코멘트  
- 운영 큐·보류 사유 상세  
- 미확정 내부 판단 표현  

---

## 7. 섹션별 출력 규칙

### 7-1. `case_overview`

**포함 예시**

- 사건 종류  
- 사건명 또는 제목  
- 기본 배경 요약  

**원칙**

- 2~4문장 이내  
- 사건을 처음 보는 사람이 빠르게 이해 가능해야 함  

### 7-2. `fact_summary`

**포함 예시**

- 핵심 사실관계  
- 시점·행위·대상 요약  
- 중요한 맥락  

**원칙**

- 사실과 추정을 혼동하지 않음  
- 입력되지 않은 사실 추가 금지  

### 7-3. `issue_summary`

**포함 예시**

- 다툼 포인트  
- 확인 필요한 핵심 쟁점  
- 법적·사실적 쟁점 후보  

**원칙**

- “확정 판단”처럼 쓰지 않음  
- 내부용과 외부용 표현 차등 적용  

### 7-4. `status_summary`

**포함 예시**

- 현재 상태값의 사용자 친화적 설명  
- 진행 중·검토 중·승인 완료 등  

**원칙**

- `CaseStatus` 의 실제 상태와 불일치 금지  
- 승인 전을 승인 완료처럼 쓰지 않음  

### 7-5. `next_step`

**포함 예시**

- 다음 검토 단계  
- 추가 정보 요청 필요 여부  
- 문서 생성·검토 예정 안내  

**원칙**

- 내부용은 더 구체적 가능  
- 외부용은 설명형 표현 사용  

### 7-6. `risk_or_missing_info`

**포함 예시**

- 누락 정보  
- 확인 필요 첨부  
- 불명확 진술  
- 해석 주의점  

**원칙**

- 내부용 중심  
- 외부용은 과도한 불안 조성 표현 금지

## 8. 질문셋 입력과의 연결

사건 요약은 [QUESTION_SET_DEFINITION.md](./QUESTION_SET_DEFINITION.md) 의 질문 키를 입력으로 사용한다.

**대표 입력 키 예시**

- `case_title`  
- `case_type`  
- `fact_summary`  
- `issue_list`  
- `timeline`  
- `requested_outcome`  
- `required_documents`  
- `has_opponent`  
- `opponent_summary`  

**원칙**

- 필수 입력 키가 비어 있으면 요약 품질이 제한될 수 있다.  
- 질문셋 분기 결과에 따라 섹션 포함 여부가 달라질 수 있다.  
- 질문 키 변경 시 사건 요약 영향도를 검토해야 한다.

## 9. 상태값과 사건 요약의 연결

| 상태 | 사건 요약 정책 |
|------|----------------|
| `CREATED` | 최소 요약만 가능, 기본 개요 중심 |
| `INTAKE_PENDING` | 누락 정보·보완 필요 사항 중심 |
| `IN_INTERVIEW` | 중간 요약 가능, 미완 정보 명시 |
| `INTERVIEW_DONE` | 구조화된 사건 요약 초안 가능 |
| `DRAFTING` | 내부 검토용 요약 강화 가능 |
| `REVIEW_PENDING` | 승인 전 검토 중심 요약 |
| `APPROVED` | 승인 상태 반영 가능, 단 의견서 대체 금지 |
| `DELIVERED` | 전달 이후 기록성 설명 중심 |
| `CLOSED` | 보관·조회 중심 |
| `HOLD` | 진행 중단 상태 명시 |
| `REJECTED` | 반려 상태와 그 의미를 설명 |
| `DELETED` | 일반 요약 출력 대상 아님 |

## 10. 내부용·외부용 차등 규칙

### 10-1. 내부용 사건 요약

**허용**

- 검토 필요 표시  
- 누락 정보 표시  
- 추가 질문 제안  
- 보완 필요 첨부 표시  
- 승인 전 상태 상세 표시  

### 10-2. 외부용 사건 요약

**허용**

- 사건 설명  
- 현재 진행 단계 설명  
- 다음 안내  
- 필요한 추가 자료 안내  

**제한**

- 내부 검토 메모  
- 승인 대기 내부 표현  
- 담당자 판단 메모  
- 미확정 쟁점의 단정 표현  

---

## 11. 승인 전·후 구분 규칙

**승인 전 사건 요약**

- 초안 또는 설명용 정리  
- 검토 전 참고용  
- 확정 판단처럼 보이면 안 됨  
- 고지문 필요  

**승인 후 사건 요약**

- 승인 상태 반영 가능  
- 승인 문서와는 별도  
- 승인 사실을 설명할 수 있으나, 문서 자체를 대체하지 않음  

**원칙**

- 사건 요약은 승인 후에도 법률 의견서 자체와 동일하지 않다.  

---

## 12. 금지 표현 규칙

사건 요약에는 아래 표현을 직접 쓰지 않는다.

- “반드시 승소”  
- “최종 법률 판단 확정”  
- “검토 완료이므로 바로 제출 가능”  
- “AI가 판단한 최종 결론”  
- “법률상 보장”  
- “문제 없음으로 확정”  

### 처리 원칙

- 금지 표현 포함 시 재생성·수정 요청  
- 후속 자동 검사 정책과 연결 가능  

---

## 13. 길이 및 표현 가이드

### 길이 가이드

- 사건 개요: 2~4문장  
- 사실관계 요약: 3~8문장  
- 쟁점 요약: 1~5개 항목 또는 2~5문장  
- 다음 단계: 1~3문장  

### 표현 가이드

- 짧고 명확하게  
- 과장 금지  
- 확정처럼 보이는 표현 금지  
- 사실과 해석을 분리  
- 외부용은 더 부드럽고 설명형으로  

---

## 14. 고지문·면책문구 연결

사건 요약에는 [NOTICE_AND_DISCLAIMER_DEFINITION.md](./NOTICE_AND_DISCLAIMER_DEFINITION.md) 기준 문구를 연결한다.

**예시 연결**

- 인터뷰 중간 요약 → “현재까지 입력 내용을 바탕으로 한 참고용 정리”  
- 인터뷰 완료 후 요약 → “문서 생성 전 구조화 초안”  
- 의뢰인용 요약 → “설명용 정리이며 최종 의견서와 동일하지 않을 수 있음”

## 15. 권한과 사건 요약의 연결

권한정의서 기준으로 사건 요약 권한은 아래처럼 본다.

| 행위 | USER | LAWYER | STAFF | ADMIN | SUPER_ADMIN |
|------|------|--------|-------|-------|-------------|
| 인터뷰 중간 요약 보기 | 제한 | O | O | O | O |
| 사건 요약 생성 | X | O | O | O | O |
| 내부 검토용 요약 보기 | X | O | O | O | O |
| 외부용 사건 요약 보기 | 자기 사건 제한 | O | O | O | O |
| 사건 요약 수정·재생성 | X | O | O(보조) | O | O |

## 16. 출력 형태

사건 요약은 아래 형태로 노출될 수 있다.

- 사건 상세 상단 카드  
- 인터뷰 완료 후 요약 패널  
- 내부 검토 패널  
- 의뢰인 설명용 문서·뷰  
- API 응답 구조체  

### 원칙

- 같은 요약이라도 노출 위치에 따라 축약·확장 정도를 달리할 수 있다.  
- 그러나 핵심 사실과 상태 설명은 서로 충돌하면 안 된다.  

---

## 17. 미확정 항목

아래는 후속 확정이 필요하다.

1. 상태별 최대 출력 길이  
2. 의뢰인용 요약 자동 생성 허용 범위  
3. 첨부자료 요약 포함 수준  
4. 요약 버전 저장 정책  
5. 내부용과 외부용 섹션 공통·차등 구조의 세분화  

---

## 18. 구현 대조 체크포인트

구현과 대조할 때는 아래를 본다.

- 사건 요약이 질문셋 입력과 연결되는가  
- 상태 설명이 실제 `CaseStatus` 와 일치하는가  
- 승인 전·후 문구가 구분되는가  
- 외부용에서 내부 메모가 숨겨지는가  
- 금지 표현이 노출되지 않는가  
- 사건 상세 카드와 내부 검토 패널 요약이 충돌하지 않는가
