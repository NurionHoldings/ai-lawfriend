## 0. 기입 원칙

- 이번 초안은 **전제 2 해소 선언용이 아니라 1차 기입용**이다.
- 상위 판정표의 **미해소** 상태를 **임의로 바꾸지 않는다.**
- **OPEN-1 ~ OPEN-5**는 이미 고정 판정표에서 **모두 잔여 유지**로 잠겨 있으므로, 이번에는 각 행의 **통제 연결 축만** 1차로 표시한다.
- `check-status --scope case` **경고 수**는 OPEN 행 판정의 **직접 근거로 쓰지 않는다.**

## 1. OPEN-1 — `APPROVED` → `CLOSED`

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| ☐ | ☐ | ☐ | ☐ |

**실무 한 줄**

`APPROVED` → `CLOSED`는 OPEN 본표에서 **유지 사유가 남아 있고** 실제 **종료 통제선**이 UI/API/서버/운영 **어디에서 확정**되는지 아직 **입증이 부족**하다.

**근거 메모**

- OPEN 5행 중 `APPROVED` → `CLOSED`는 의미상 **종료 흐름**과 맞닿아 있어 운영·감사 축도 **—**보다는 **☐**가 적절하다.
- 다만 현재는 `notes` 보강만으로 닫힌 행이 아니고, **종료 액션**과 OPEN 유지 사유의 관계가 완전히 잠겼다고 보기 어렵다.
- 따라서 **4축 모두** 추가 근거 확인 전까지 **☑ 없이** 유지한다.

## 2. OPEN-2 — `REJECTED` → `CREATED`

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| ☐ | ☐ | ☐ | — |

**실무 한 줄**

`REJECTED` → `CREATED`는 **[175] 정합안 A**와 직접 충돌하지 않도록 `notes`가 정리됐지만, **직접 재진입**을 실제 통제선에서 어떻게 다루는지 근거가 부족해 UI/API/서버는 **미입증** 상태다.

**근거 메모**

- 정리한 반영 내용상 **OPEN-2**가 `REJECTED` → `CREATED` 행이며, 원안 OPEN-3 문구가 이 행으로 재배치되었다([EVIDENCE-20260421-184]).
- 현재는 **문서 의미 정렬**은 되었지만, 실제 제품에서 이를 **허용/차단/비전이 처리** 중 무엇으로 다루는지 **확정 근거가 부족**하다.
- 운영·감사는 이 행에서 **직접 통제 축으로 보기 어려워** 우선 **—** 처리한다.

## 3. OPEN-3 — `HOLD` → `HOLD`

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| ☐ | — | — | ☐ |

**실무 한 줄**

`HOLD` → `HOLD`는 **상태 유지** 성격의 메타 행에 가까워 API/서버 **전이 통제**보다는 **화면상 유지 표시**와 **운영상 보류 추적**이 있는지부터 확인이 필요하다.

**근거 메모**

- `HOLD` → `HOLD`는 전형적인 허용 전이보다 **상태 유지·보류 관리** 성격이 강하므로 API/서버를 곧바로 **상태 전이 통제선**으로 보기 어렵다.
- 대신 UI에서 보류 상태 **액션 노출/비노출**과 운영·감사에서 **보류 사유 추적**이 있는지가 더 중요하다.
- 1차안은 **UI ☐ / API — / 서버 — / 운영·감사 ☐**로 둔다.

## 4. OPEN-4 — `DELETED` → `CREATED`

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| ☐ | ☐ | ☐ | ☐ |

**실무 한 줄**

`DELETED` → `CREATED`는 현재 기준에서 **직접 허용 전이 아님**으로 읽고, 문서상으로는 **DENY-8**이 일반 재진입 금지·**OPEN-4**가 복원·재생성·삭제 후 처리 설명 축을 맡는 구조로 유지한다 ([227]). UI/API/서버/운영 **전 축**에서 실제 허용·차단·감사 흔적은 여전히 **추가 확인**이 필요하다.

**근거 메모**

- 삭제 이후 생성 축은 **복원, 재생성, soft delete** 처리와 쉽게 엮인다.
- 단순 문서 `notes`만으로는 닫기 어렵고, UI/API/서버/운영 전 축에 모두 통제 연결 가능성이 있다.
- 특히 soft delete 또는 복원 흐름과의 관계가 남아 있으므로 **4축 모두 ☐**가 적절하다.
- **[228]:** `softDeleteCaseService`가 권한 후 `updateCaseById`→`DELETED`로 **진입** 확인. **상태별 삭제 제한**·`DELETED`→`CREATED` 직접 복구는 [228] 스코프에서 미확인 — [227]과 확인 범위에서 정합.

**교차 정리에서 OPEN-4 측에 고정한 삽입 문안 ([227]):**

- **OPEN-4**는 `DELETED` → `CREATED`를 **직접 허용 전이로 확정하는 행이 아니다.**
- 본 행은 **DENY-8**의 일반 재진입 금지와 충돌하지 않도록, 삭제 이후 처리 경로를 보수적으로 설명·추적하는 축이다.
- 후속 정리는 복구 정책, 운영 기준, 실제 구현 경로, 통제 근거 확인과 함께 **별도 증빙**으로 누적한다.

## 5. OPEN-5 — 메타

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| — | — | — | ☐ |

**실무 한 줄**

OPEN **메타** 행은 개별 전이보다는 **유지 사유·해제 조건·재판정 추적**을 다루므로 **운영·감사 축** 확인이 우선이며, 나머지 축은 직접 관련이 약하다.

**근거 메모**

- 정리한 구조상 OPEN-5는 **메타** 행이다.
- 메타 행은 개별 버튼/라우트/서비스보다 **운영 기준, 재판정, 감사 흔적**과 연결되는 성격이 강하다.
- 따라서 **운영·감사 ☐**를 두고 나머지는 일단 **—** 처리하는 것이 가장 자연스럽다.

## 7. 이 초안의 해석 규칙

- 이 표는 **OPEN 5행 전부 미해소 유지**를 전제로 한다.
- ☐가 많다는 이유만으로 곧바로 **구현 오류**라고 단정하지 않는다.
- 다만 현재 시점에는 OPEN 5행의 **통제 연결이 충분히 입증되지 않았다**는 뜻이다.
- 따라서 이 초안은 다음 단계의 **실제 코드·화면·운영 흐름** 확인을 위한 **출발점**이다.

## 8. 문서에 붙일 한 줄 결론

§11 OPEN-1 ~ OPEN-5 1차 기입은 **모두 미해소 유지** — 문서 의미 정렬은 되었지만, 각 행의 **유지 사유·해제 조건**이 실제 제품 통제선(UI·API·서버·운영) **어디에 연결되는지**는 아직 **추가 입증**이 필요하다.
