# AI법친 핵심 흐름 마감 점검표

**용도:** 본 작업 **우선순위 1번** — 공표 전 기준으로 **핵심 사용자 흐름**이 **실제 구현·정의서·운영 설명**까지 끊김 없이 닫혔는지 확인한다.  
**관련:** DELETED·repository 축 해석은 증빙 **[243]** (`IMPLEMENTATION_EVIDENCE.md`) · 사건 상세 점검 시 **raw vs service** 불균일 등을 참고할 수 있다.

---

## 0. 점검 기준

### 점검 원칙

- **새 기능 추가**보다 **기존 핵심 흐름 마감**을 우선한다.
- **정의서와 충돌**하면 **구현보다 정의서** 기준으로 맞춘다.
- **“동작함”만**으로 완료 처리하지 않고, 아래까지 닫혀야 **완료**로 본다.  
  **권한** · **상태값** · **사용자 설명** · **운영 처리**

### 점검 상태값

| 상태 | 의미 |
|------|------|
| **완료** | 구현 / 문서 / 설명 / 운영 처리까지 닫힘 |
| **보정 필요** | 핵심은 되지만 문구·분기·권한·예외 처리 보강 필요 |
| **미완료** | 핵심 흐름이 아직 끊김 |
| **보류** | 지금 우선순위 아님 |

## 1. 변호사 가입 / 승인

### 점검 항목

- 변호사 가입 화면이 실제로 동작하는가
- 가입 후 상태가 승인 전/후로 명확히 구분되는가
- 관리자 승인 없이는 핵심 기능 접근이 제한되는가
- 승인/반려 후 사용자에게 상태 안내가 되는가
- 정의된 권한 구조와 실제 계정 역할이 일치하는가

### 확인 질문

- 승인 전 계정이 **어디까지** 접근 가능한가
- 승인 후 **role/state**가 **어디서** 확정되는가
- 관리자 화면에서 승인 처리 흐름이 **끊기지 않는가**

### 완료 기준

가입 → 승인 대기 → 관리자 승인 → 로그인 가능 흐름이 **끊김 없이** 닫힌다.

| **근거** | |

## 2. 로그인

### 점검 항목

- 로그인 성공/실패 처리가 정상인가
- role별 첫 진입 화면이 자연스러운가
- 비로그인 사용자가 보호 페이지 접근 시 제대로 차단되는가
- 세션 만료 시 처리 문구가 명확한가
- 로그아웃 후 보호 페이지 재접근이 막히는가

### 확인 질문

- 관리자 / 변호사 / 스태프 / 의뢰인별 **진입점**이 맞는가
- 보호 레이아웃과 실제 **API 보호**가 같이 맞물리는가

### 완료 기준

로그인 / 로그아웃 / 세션 보호가 **역할별로** 안정적이다.

## 3. 대시보드

### 점검 항목

- 로그인 직후 사용자가 봐야 할 **핵심 정보**가 보이는가
- 역할별로 대시보드 **구성**이 맞는가
- 사건 목록, 최근 작업, 경고/알림, 다음 할 일이 **자연스럽게** 이어지는가
- 빈 상태(empty state) 문구가 있는가
- 운영상 중요한 카드가 누락되지 않았는가

### 확인 질문

- 변호사/관리자/의뢰인 대시보드가 **같은 화면**이어도 괜찮은가
- 공표 시 보여줄 **핵심 가치**는 대시보드에서 바로 드러나는가

### 완료 기준

**“로그인 후 무엇을 해야 하는지”**가 대시보드에서 **바로** 보인다.

## 4. 의뢰인 초대 / 가입

### 점검 항목

- 사건과 의뢰인 **연결 방식**이 명확한가
- 초대 링크 또는 가입 흐름이 실제로 이어지는가
- 잘못된 초대/만료/중복 상황 **처리 문구**가 있는가
- 의뢰인 계정 생성 후 볼 수 있는 **화면이 제한**되는가
- 담당자와 의뢰인의 **권한 차이**가 명확한가

### 확인 질문

- 의뢰인이 **어디서** 사건에 연결되는가
- 초대 후 **최초 진입** 화면이 자연스러운가

### 완료 기준

초대 → 가입 → 사건 연결 → 기본 사용 흐름이 **닫힌다**.

## 5. 사건 생성

### 점검 항목

- 사건 생성 폼 **필수값/선택값**이 정의서와 일치하는가
- 생성 후 **상태**가 올바르게 시작되는가
- 생성 직후 **어디로 이동**하는가
- 감사로그 / 타임라인 / 기본 배너가 맞게 붙는가
- 빈 값, 잘못된 값 **예외 처리**가 되는가

### 확인 질문

- 생성 직후 상태가 **CREATED**인지, 혹은 **현재 기준 상태값**과 일치하는가
- 이후 **인터뷰**로 자연스럽게 이어지는가

### 완료 기준

사건 생성이 후속 흐름(인터뷰/보완/문서)과 **자연스럽게** 연결된다.

## 6. AI 인터뷰

### 점검 항목

- 질문셋 **로딩**이 실제로 맞는가
- **조건 분기**가 제대로 작동하는가
- 답변 **저장**이 안정적인가
- 인터뷰 **완료 처리** 후 상태 전이가 올바른가
- 인터뷰 중단/재개/예외 상황이 **설명**되는가

### 확인 질문

- 질문셋 **정의서**와 실제 인터뷰 흐름이 일치하는가
- 완료 후 사건 요약/문서 생성으로 이어질 **준비**가 되는가

### 완료 기준

질문 → 답변 → 조건 분기 → 저장 → 완료 처리까지 **닫힌다**.

## 7. 사건 요약

### 점검 항목

- 인터뷰 기반 요약이 **실제로 생성**되는가
- 사건 상세 상단이나 **핵심 위치**에 노출되는가
- 요약이 사용자 **역할**에 맞게 이해 가능한가
- 재생성/갱신 **규칙**이 명확한가
- 요약과 **원문 데이터**의 연결이 유지되는가

### 확인 질문

- 요약은 **참고 정보**인지, **실무 기준 정보**인지 **구분**되는가
- 공표 시 보여줄 **가치 포인트**가 충분한가

### 완료 기준

인터뷰 후 사건 요약이 **자연스럽게** 사용자 흐름에 편입된다.

## 8. 문서 생성

### 점검 항목

- 인터뷰 완료 후 **문서 생성 진입**이 가능한가
- 질문셋 / 템플릿 / 문서 타입 **연결**이 맞는가
- 생성 **실패** 시 이유가 분명한가
- 생성 후 **문단 구조**와 **버전** 흐름이 이어지는가
- DELETED / REJECTED / CLOSED 같은 **예외 상태 차단**이 맞는가 (증빙 **[243]**·`documents/generate` 실사 정합)

### 확인 질문

- 문서 생성 **라우트**와 **화면**이 실제 사용자 경험상 자연스러운가
- 생성 후 **검토/승인/잠금**으로 이어지는가

### 완료 기준

문서 생성이 성공/실패/예외 상태 포함해 **안정적으로** 닫힌다.

> **실제 확정 판정(6축·파일 순서·실무 표):** [CASE_DETAIL_ACTUAL_JUDGMENT_V1.md](./CASE_DETAIL_ACTUAL_JUDGMENT_V1.md) — 확정 기준(4상태)과 **1차 예상(에테르니언)** 은 `§5`~`§7` 를 본다.

### 점검 항목

- 상태 카드 / 진행 액션 / 문서 목록 / 타임라인 / 요약 / 문단 패널이 **유기적으로** 연결되는가
- 역할별로 **보이면 안 되는 버튼**이 숨겨지는가
- 삭제/종결/반려/보류 같은 **상태**에서 UI가 적절히 달라지는가
- 상세 화면이 실제 **업무 허브** 역할을 하는가
- **라우트별** raw 조회와 service 경유 **차이**가 운용·공표 기준에서 **수용 가능한가** (참고: 증빙 **[243]** `detail` API vs `getCaseAccessContext` 경로)

### 확인 질문

- 상세 화면 **하나**로 실무 대부분을 처리할 수 있는가
- 현재 가장 **혼란스러운** 버튼/동작은 무엇인가

### 완료 기준

사건 상세가 **업무 중심 화면**으로 자연스럽게 작동한다.

## 10. 보완 요청

### 점검 항목

- 접수 보완 / 검토 대기 / 보완 안내 **흐름**이 사용자에게 이해되게 보이는가
- 의뢰인과 담당자에게 **다른 설명**이 제공되는가
- 보완 요청 후 다시 사건 상세/인터뷰/자료 제출로 **잘 이어지는가**
- 안내 배너/허브/링크 **구조**가 실제로 잘 작동하는가

### 확인 질문

- 보완 요청은 **“기능”**으로도 닫혔는가, 아니면 아직 **안내 문구** 중심인가
- 의뢰인이 **혼란 없이** 다음 행동을 알 수 있는가

### 완료 기준

보완 요청이 단순 문구가 아니라 **실제 사용자 행동 흐름**으로 닫힌다.

## 11. 관리자 승인

### 점검 항목

- 승인 **로그/이력/표시**가 남는가

### 완료 기준

관리자 승인 흐름이 문서/사건/상태와 함께 **완결**된다.

## 13. 마감 판정 기준

### 공표 가능

- 핵심 흐름 **11개** 중 **대부분**이 **완료**
- 나머지도 **보정 필요** 수준
- **미완료**가 핵심 흐름을 **끊지 않음**

### 공표 전 보강 필요

- **사건 생성 → 인터뷰 → 문서 생성 → 사건 상세 → 승인**  
  이 **주 흐름** 중 **하나라도** 끊김

### 내부 테스트 단계 유지

- 가입/로그인/권한/상태 전이 중 **핵심**이 아직 **불안정**

**AI법친 본 작업의 우선순위 1번**은, 핵심 사용자 흐름 **11개**를 기준으로, 이미 잠근 **정의서**와 **현재 구현**을 계속 **정렬**하면서 **공표 가능한 수준**까지 **마감**하는 것이다.

## 15. 부록 — 1차 예상 판정 (기억 기준 / 실사 전 임시)

<a id="appendix-1st-judgment"></a>

**전제 (필독):** 본 절의 상태·근거·조치는 **현재 기억·세션 맥락**에 따른 **예상치**이며, **실레포 실사** 이전 **임시 판정**이다. **공표·완료 선언용 확정 판정이 아니다.** 불확실성을 줄이기 위해 **보수적**으로 잡았다. 실제 점검 후 `§12`·본 절·`§1`~`§11` **현재 판정 칸**을 **덮어쓰기** 권장.

### 1. 변호사 가입 / 승인

- **상태:** 보정 필요  
- **근거:** 가입/승인 흐름은 여러 차례 우선순위 핵심으로 잡혔고, 권한 구조도 중요하게 다뤄졌지만, 최근 세션의 핵심 작업은 DELETED 축 실사와 거버넌스 정렬이어서 “공표 직전 수준으로 매끈하게 닫혔다”는 **확정 근거는 아직 부족**하다.  
- **보정 필요:** 승인 전/후 사용자 노출 화면, 승인 대기 문구, 관리자 승인 처리 UX 최종 점검  
- **다음 조치:** 변호사 가입 → 승인 대기 → 관리자 승인 → 로그인 가능까지 **실제 화면/API 1회 종주** 점검  

### 2. 로그인

- **상태:** 완료에 가까운 보정 필요  
- **근거:** 인증 세트, 보호 레이아웃, 세션 사용자 확인, 역할별 접근 구조는 이미 여러 차례 구현 축으로 다뤄졌고 **비교적 안정적인 기반**으로 본다.  
- **보정 필요:** 역할별 첫 진입 화면 자연스러움, 세션 만료/오류 문구 마감  
- **다음 조치:** 관리자/변호사/의뢰인 **각 1회** 로그인·로그아웃·보호 페이지 재접근 점검  

### 3. 대시보드

- **상태:** 보정 필요  
- **근거:** 대시보드 관련 기능은 존재하지만, 공표 관점에서 “로그인 후 **무엇을 해야 하는지**”가 **명확히 보이는 수준**까지 정리되었는지는 추가 점검이 필요하다.  
- **보정 필요:** 역할별 핵심 카드, 다음 할 일, 빈 상태 문구, 최근 작업 요약  
- **다음 조치:** 관리자/변호사 기준 **첫 화면 정보 구조** 점검 후 **카드 우선순위** 정리  

### 4. 의뢰인 초대 / 가입

- **상태:** 보정 필요  
- **근거:** MVP 흐름상 필수 축으로 기억되어 있으나, 최근 세션에서 **직접 실사한 축은 아님**. 실제 초대 링크/가입 후 **사건 연결 UX**를 다시 확인해야 한다.  
- **보정 필요:** 초대 만료/오류/중복 처리, 의뢰인 첫 진입 화면, 사건 연결 직후 안내  
- **다음 조치:** 초대 → 가입 → 사건 연결 → 첫 화면까지 **시나리오 점검**  

### 5. 사건 생성

- **상태:** 완료에 가까운 보정 필요  
- **근거:** 사건 생성, 사건 목록, 사건 상세, 상태값, soft delete까지 **계속 실사**되었고 **기본 CRUD 축은 상당히 안정**된 것으로 본다.  
- **보정 필요:** 생성 직후 이동 경로, 필수/선택값 문구, 생성 후 다음 행동 안내  
- **다음 조치:** 사건 생성 폼 입력 → 생성 후 **상세/인터뷰 유도** 흐름 확인  

### 6. AI 인터뷰

- **상태:** 보정 필요  
- **근거:** 질문셋 기반 인터뷰, 조건 분기, 인터뷰 완료 처리, 상태 전이 등은 **상당히 구현이 앞서간다**고 잠겨 있으나, **사용자 경험**과 **예외 처리** 마감은 재점검 필요.  
- **보정 필요:** 인터뷰 중단/재개, 답변 저장 확인, 완료 후 사건 요약/문서 생성 **연결 문구**  
- **다음 조치:** 질문 **분기 포함** 실제 인터뷰 1회 종주 + **완료 후 후속 화면** 연결 확인  

### 7. 사건 요약

- **상태:** 보정 필요  
- **근거:** 사건 요약은 핵심 흐름에 포함, 상세 상단 연결도 작업 기억에 남아 있으나, 공표용 데모 관점에서 “**얼마나 명확하게 보이는지**”는 추가 정리 필요.  
- **보정 필요:** 요약 카드 위치, 재생성/갱신 규칙, 사용자 설명  
- **다음 조치:** 인터뷰 완료 후 **요약 생성/표시/갱신** 흐름 점검  

### 8. 문서 생성

- **상태:** 완료에 가까운 보정 필요  
- **근거:** `documents/generate` **실사**까지 마쳤고, 인터뷰 완료 후 생성, **상태 차단**, 템플릿 연결 등 **핵심 구조가 상당히 선명** (증빙 **[243]**).  
- **보정 필요:** 생성 실패 메시지, 사용자에게 보이는 **진입 버튼/상태 설명**, 생성 후 **다음 단계** 안내  
- **다음 조치:** 인터뷰 완료 사건 기준 **문서 생성 성공/실패/예외** 상태 점검  

### 9. 사건 상세

- **상태:** 보정 필요  
- **근거:** 이번 DELETED 축 실사에서 **가장 많이 확인된 화면** 중 하나로 구조는 꽤 선명하나, **raw 상세 / API / service** 경로 **불균일**이 드러났기에 **공표 전 실사용 중심 화면**으로 마감이 필요.  
- **보정 필요:** 버튼 노출 일관성, 역할별 안내, **진행 액션** 우선순위, 혼란 요소 제거  
- **다음 조치:** 변호사/의뢰인 기준 **사건 상세 화면 종합 UX** 점검  

### 10. 보완 요청

- **상태:** 보정 필요  
- **근거:** 보완 안내 허브, 접수 보완/검토 대기 배너, 역할별 안내는 **꽤 정리**된 기억이 있으나, **실제 사용자 행동 흐름**으로 완전히 닫혔는지는 **한 번 더** 볼 필요.  
- **보정 필요:** 의뢰인/담당자 **문구 차등**, “다시 어디로 가야 하는지” **행동 유도**, 링크 일관성  
- **다음 조치:** **접수 보완** 사건 / **검토 대기** 사건 **각 1개씩** 시나리오 점검  

### 11. 관리자 승인

- **상태:** 보정 필요  
- **근거:** 승인, 잠금, 문서 승인본, 검증코드, 승인 흐름 관련 **구현·설계**가 **많이 진척**된 기억이 있으나, **공표 가능한 관리자 UX**로 닫혔는지는 **마지막 점검**이 필요.  
- **보정 필요:** **승인 큐** 경험, 승인/반려 후 **사용자 측 상태** 표시, **승인 로그** 가시성  
- **다음 조치:** **관리자 승인 시나리오 1회 종주** + **승인 후 사용자 화면** 확인  

### 부록-요약: 한눈에 쓰는 요약 (§12와의 관계)

`§12` **최종 종합 점검표 (요약판)** 본문과 **동일**하다. (부록만 인쇄·공유할 때를 위해 이 문장을 둔다.)

### 부록: 1차 전체 판정 (에테르니언 기준)

| 구분 | 현재 **예상** |
|------|---------------|
| **완료** | (보수적으로) **없음** — 아직 |
| **완료에 가까운 보정 필요** | 로그인, 사건 생성, 문서 생성 |
| **보정 필요** | 나머지 대부분 |
| **미완료** (기억 기준) | 핵심 흐름 **자체**가 **완전히 비어 보이는 축**은 **없음**으로 본다 |

**한 줄 요약:** 핵심 기능은 **상당수 연결**되어 있으나, **공표 직전** 관점에서는 **사용자 흐름 마감**과 **설명·예외 처리 보정**이 **필요한 단계**로 본다.

### 부록: 지금 가장 먼저 손대야 할 3개 (공표 기준 1차 Top 3)

1. **사건 상세** — 가장 많은 기능이 모여 **실제 업무 허브**이므로.  
2. **AI 인터뷰 → 사건 요약 → 문서 생성** — AI법친 **가치가 가장 잘 드러나는** 핵심 연결.  
3. **관리자 승인** — 실무 **마감**과 **신뢰성**을 보여주는 **마지막 축**이므로.  

### 부록: 바로 다음 추천 작업 (실무 절차)

11개를 **한꺼번에** 끝까지 보기보다, **「사건 상세 → (AI 인터뷰 / 사건 요약 / 문서 생성) → 관리자 승인」** **3축**부터 **실제 상태**를 **먼저 확정**하는 것이 좋다. 확정되면 `§12`·`§1`~`§11` 판정 칸·본 부록의 **1차** 문구를 **대체**한다.
