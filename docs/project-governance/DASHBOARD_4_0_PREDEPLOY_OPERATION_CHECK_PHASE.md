# 대시보드 4.0 — 배포 전 운영 점검 Phase 착수 문서

- `docs/project-governance/DASHBOARD_3_11_FINAL_SEAL_SUMMARY.md`
  - §12: 대시보드 3.x 최종 확인 순서
  - §13: 대시보드 3.x 봉인 이후 진행 기준

## 1. 문서 목적

이 문서는 대시보드 3.x 최종 봉인 이후, 배포 전 운영 점검을 별도 Phase로 분리하기 위한 착수 문서다.

대시보드 4.0은 신규 기능 개발 Phase가 아니다.  
대시보드 4.0은 배포 전 운영자가 확인해야 할 접근, 권한, 표시, 문구, 증빙, 회귀 점검 항목을 정리하는 운영 점검 Phase다.

## 2. 3.x 봉인 유지 원칙

대시보드 3.x는 아래 기준으로 봉인 상태를 유지한다.

- `DASHBOARD_3_11_FINAL_SEAL_SUMMARY.md` §12~13을 최종 기준으로 본다.
- `#evidence-20260426-391`을 대시보드 3.x 최종 봉인 증빙으로 본다.
- 3.x 범위에서 기능 확장을 하지 않는다.
- 3.x 증빙을 임의로 재작성하지 않는다.
- 3.x에서 닫은 metric, badge, 문구, demo metrics 경계를 재오픈하지 않는다.

## 3. QA closure 미기입 유지

아래 항목은 QA 팀 회신 전까지 작성하지 않는다.

- `#evidence-20260428-predeploy-qa-closure`의 확정 기록 표
- QA 팀 회신 원문 줄
- QA 최종 통과 판정
- 배포 가능 최종 판정

QA 회신 수신 후에만 실제 회신 내용 기준으로 작성한다.

## 4. 대시보드 4.0 범위

대시보드 4.0에서 할 수 있는 일:

- 배포 전 운영 점검 항목 정리
- 역할별 접근 확인 항목 정리
- 권한별 화면 진입 확인 항목 정리
- 빈 상태 / 로딩 / 오류 표시 확인 항목 정리
- 사용자 노출 문구 확인 항목 정리
- 금지 표현 재점검 항목 정리
- 증빙 문서 확인 순서 정리
- QA 회신 수신 전 준비 가능한 운영 점검표 작성

대시보드 4.0에서 하지 않는 일:

- 신규 API route 생성
- DB schema 변경
- CaseStatus canonical 변경
- dashboard metric 계산 변경
- badge 계산 변경
- DashboardPreviewCard 구조 변경
- 실제 알림·에스컬레이션 로직 추가
- 배포 전 QA closure 확정표 작성
- 대시보드 3.x 기능 재오픈

## 5. 배포 전 운영 점검 대분류

대시보드 4.0의 점검 항목은 아래 대분류로 정리한다.

1. 접근 / 권한 점검
2. 역할별 대시보드 렌더링 점검
3. 실데이터 표시 점검
4. 빈 상태 / 로딩 상태 점검
5. 문구 / 금지 표현 점검
6. demo metrics 경계 점검
7. 증빙 / 내비게이터 연결 점검
8. QA closure 대기 상태 점검

## 6. 1차 체크리스트

### 6.1 접근 / 권한 점검

- [ ] 의뢰인 권한 사용자는 `/dashboard`에 접근할 수 있다.
- [ ] 변호사 권한 사용자는 `/lawyer`에 접근할 수 있다.
- [ ] 관리자 권한 사용자는 `/admin`에 접근할 수 있다.
- [ ] 권한이 맞지 않는 사용자는 restricted 안내 경로로 처리된다.
- [ ] Living Logo V2 restricted 상태와 충돌하지 않는다.

### 6.2 역할별 대시보드 렌더링 점검

- [ ] 의뢰인 대시보드가 정상 렌더링된다.
- [ ] 변호사 대시보드가 정상 렌더링된다.
- [ ] 관리자 대시보드가 정상 렌더링된다.
- [ ] DashboardPreviewCard가 각 역할에서 깨지지 않는다.
- [ ] 모바일 / 좁은 화면에서 주요 카드가 겹치지 않는다.

### 6.3 실데이터 표시 점검

- [ ] 의뢰인 대시보드는 의뢰인 실데이터 기준으로 표시된다.
- [ ] 변호사 대시보드는 변호사 실데이터 기준으로 표시된다.
- [ ] 관리자 대시보드는 관리자 실데이터 기준으로 표시된다.
- [ ] demo metrics가 실서비스 경로에 섞이지 않는다.
- [ ] `dashboard-demo-metrics.ts`는 실서비스 집계 기준으로 사용되지 않는다.

### 6.4 빈 상태 / 로딩 상태 점검

- [ ] 사건이 없는 의뢰인 화면에서 빈 상태가 자연스럽게 표시된다.
- [ ] 검토할 사건이 없는 변호사 화면에서 빈 상태가 자연스럽게 표시된다.
- [ ] 운영 확인 후보가 없는 관리자 화면에서 빈 상태가 자연스럽게 표시된다.
- [ ] 장기 미진행 후보가 없는 경우 보조 문구가 자연스럽게 표시된다.
- [ ] 로딩 상태가 있는 화면은 사용자에게 과장된 표현 없이 표시된다.

### 6.5 문구 / 금지 표현 점검

- [ ] 승소 가능성 표현이 없다.
- [ ] 패소 가능성 표현이 없다.
- [ ] 법률 위험도 표현이 없다.
- [ ] 장애 / 치명 / P0 / P1 표현이 없다.
- [ ] 운영 사고 / 위험 폭증 표현이 없다.
- [ ] 에스컬레이션 표현이 실제 로직처럼 보이지 않는다.

### 6.6 증빙 / 내비게이터 점검

- [ ] `DASHBOARD_3_11_FINAL_SEAL_SUMMARY.md` §12~13이 최신 기준이다.
- [ ] `IMPLEMENTATION_EVIDENCE.md`의 `#evidence-20260426-391`이 최신 3.x 봉인 증빙이다.
- [ ] `tools/aibeopchin_navigator.py`에 `dashboard_3_11_final_seal_summary`가 유지된다.
- [ ] 4.0 문서는 3.x 재오픈이 아니라 별도 Phase로 표시된다.
