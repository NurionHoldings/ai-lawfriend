## 4축 정의 (행마다 동일 해석)

| 축 | 점검 내용(요지) |
| --- | --- |
| **UI** | 해당 잔여와 관련된 **화면·버튼·드롭다운·가드**에서 금지·미합의 전이가 **사용자에게 노출·실행 불가**한가. |
| **API** | **HTTP 라우트·입력 스키마·권한**에서 요청 자체가 거절되거나, 허용 목록과 불일치 시 **일관되게 실패**하는가. |
| **서버** | **`applyCaseStatusTransition`** · 도메인 서비스 · DB 제약 등 **서버 단 검증**으로 잔여·금지가 지켜지는가. |
| **운영·감사** | **로그·알림·운영 규칙·감사 추적**으로 위반·애매 전이가 식별 가능한가(해당 시). |

**셀 기입 규칙 (권장)**

- `☐` 미점검 · `☑` 충족 · `—` 해당 없음(근거 한 줄을 **점검 메모**에 적는다).
- **1차 기입** 시 상세 규칙·4축 판정·우선순위·금지 사항은 [PREMISE2_CONTROL_CONNECTION_AUDIT_FIRST_PASS_CRITERIA.md](./PREMISE2_CONTROL_CONNECTION_AUDIT_FIRST_PASS_CRITERIA.md) 를 따른다.
- 최소 **한 축**은 `☑` 또는 “의도적 비적용(근거 있음)”이어야 [CASE_LIFECYCLE_TRANSITION_SUMMARY.md](./CASE_LIFECYCLE_TRANSITION_SUMMARY.md) 해소 조건표 §2 항 4 해소 후보로 본다(1차 기입 최소 통과 기준은 기준표 §6 참조).
