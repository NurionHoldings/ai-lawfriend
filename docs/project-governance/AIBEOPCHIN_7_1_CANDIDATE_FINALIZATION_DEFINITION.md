# AI법친 7.1 — 후보 확정 정의서

## 1. 문서 목적

AI법친 7.1 후보 확정 정의서는 7.0 운영 모니터링 대시보드에서 분류한 운영 피드백과 개선 후보를 바탕으로, 7.1에서 실제 설계·구현 대상으로 삼을 후보를 확정하기 위한 기준문서다.

이 문서는 기능 구현 문서가 아니다.  
이 문서는 7.1 착수 전 후보의 범위, 우선순위, 제외 항목, 구현 전제, 증빙 기준을 확정하기 위한 문서다.

## 2. 현재 전제

AI법친 6.x 사건 패키지 기능군은 운영 배포 및 Smoke Test 14/14 PASS 이후 최종 잠금 완료 상태다.

현재 전제:

- 6.x 운영 배포 완료
- 6.x Smoke Test 14/14 PASS
- 6.x 최종 잠금 완료
- 6.x 신규 기능 추가 금지
- 7.x는 6.x와 분리된 별도 로드맵
- 7.0 운영 모니터링 대시보드 골격 완료
- 7.0 관리자 메뉴 링크 완료
- 7.0 KPI 카드 완료
- 7.0 운영 경고 상세 패널 완료
- 7.0 Runbook / 장애 대응 체크리스트 패널 완료
- 7.0 운영 상태 타임라인 / 최근 증빙 이력 패널 완료
- 7.0 운영 피드백 수집 패널 / 7.1 후보 분류 패널 완료

## 3. 7.1 후보 확정 원칙

7.1 후보는 다음 원칙에 따라 확정한다.

1. 6.x 최종 잠금 상태를 침범하지 않는다.
2. 6.x 사건 패키지 공유 정책을 임의로 변경하지 않는다.
3. DB schema 변경이 필요한 항목은 별도 7.1 DB 정의서 이후에만 착수한다.
4. API 추가가 필요한 항목은 별도 7.1 API 명세서 이후에만 착수한다.
5. AI가 최종 법률 판단을 하도록 만들지 않는다.
6. 변호사의 최종 검토·확정권을 유지한다.
7. 개인정보·보안·변호사법 고지 정책을 약화하지 않는다.
8. 운영 피드백은 즉시 6.x 수정이 아니라 7.1 후보로 분류한다.
9. 후보 확정 전에는 코드 구현을 시작하지 않는다.
10. 후보 확정 후에도 구현은 별도 패치 단위로 분리한다.

| 7.1-B | 보완 요청 워크플로우 | 상 | 의뢰인 재입력 흐름 개선 | 후보 |
| 7.1-C | 운영 로그 / 보안 감사 리포트 | 중 | 운영자 감사·보안 확인 | 후보 |

## 5. 후보별 상세 정의

### 5.1 7.1-A 변호사 검토 메모 고도화

#### 목적

변호사가 사건 패키지를 검토하면서 주요 검토 포인트, 보완 필요사항, 내부 메모를 구조화하여 남길 수 있도록 한다.

#### 기대 효과

- 변호사 검토 흐름 정리
- 사건 패키지 검토 품질 향상
- 보완 요청 전 내부 판단 정리
- 의뢰인에게 전달할 내용과 내부 검토 메모 분리

#### 포함 가능 범위

- 변호사 전용 검토 메모 영역
- 메모 유형 분류
- 검토 포인트 체크리스트
- 보완 요청 후보 표시
- 내부용 / 의뢰인 공유용 구분
- 메모 작성자 / 작성 시각 표시

#### 제외 범위

- AI의 최종 법률 판단
- 승소·패소 가능성 단정
- 자동 수임 판단
- 의뢰인에게 변호사 검토 없이 법률 결론 제공
- 기존 6.x 사건 패키지 공유 정책 변경

#### 선행 필요 문서

- 7.1-A 변호사 검토 메모 정의서
- 7.1-A 데이터 구조 정의서
- 7.1-A API 명세서
- 7.1-A 화면 정의서

### 5.2 7.1-B 보완 요청 워크플로우

#### 목적

변호사가 사건 패키지 검토 후 의뢰인에게 추가 자료, 추가 답변, 누락 정보 보완을 요청하고, 의뢰인이 다시 입력할 수 있는 흐름을 만든다.

#### 기대 효과

- 의뢰인 재입력 흐름 명확화
- 사건 패키지 완성도 향상
- 변호사 상담 전 누락 정보 최소화
- 보완 요청 이력 관리 가능

#### 포함 가능 범위

- 보완 요청 생성
- 요청 유형 분류
- 의뢰인 보완 입력
- 보완 요청 상태값
- 보완 완료 처리
- 보완 요청 이력 표시
- 보완 요청과 사건 패키지 연결

#### 제외 범위

- 변호사 검토 없는 자동 보완 요구 확정
- AI의 강제 질문 발송
- 사건 상태값 무단 변경
- 6.x 공유 만료·취소 정책 변경
- 의뢰인에게 법률 결론 직접 제공

#### 선행 필요 문서

- 7.1-B 보완 요청 워크플로우 정의서
- 7.1-B 상태값 정의서
- 7.1-B 데이터 구조 정의서
- 7.1-B API 명세서
- 7.1-B 화면 정의서

---

### 5.3 7.1-C 운영 로그 / 보안 감사 리포트

#### 목적

운영자가 사건 패키지 공유, 열람, 다운로드, 차단, 만료, 취소 이력을 더 쉽게 검토할 수 있도록 감사 리포트 형태로 정리한다.

#### 기대 효과

- 운영 보안 확인 강화
- 열람·다운로드 이력 추적 강화
- 반복 차단 접근 탐지
- 공유 위험 상태 식별
- 개인정보 보호 운영 증빙 강화

#### 포함 가능 범위

- 열람 로그 요약
- 다운로드 로그 요약
- 차단 로그 요약
- 만료 / 취소 공유 리포트
- 위험 공유 후보 표시
- 관리자 감사용 CSV 또는 화면 리포트 후보

#### 제외 범위

- 첨부파일 직접 URL 노출
- 관리자 대리 다운로드
- 공유 권한 강제 완화
- 원문 accessToken / optionalPin / hash 노출
- DATABASE_URL, storagePath 등 민감값 노출

#### 선행 필요 문서

- 7.1-C 운영 로그 / 보안 감사 리포트 정의서
- 7.1-C 로그 출력 정책서
- 7.1-C 개인정보 마스킹 기준서
- 7.1-C 화면 정의서

- 사건 요약 섹션 구조 개선
- 첨부자료 목록 표현 개선

- PDF 출력 헤더 / 푸터 개선
- 출력물 버전 표시

## 11. 최종 판정

AI법친 7.1의 1차 확정 후보는 "보완 요청 워크플로우"로 한다.

7.1은 6.x 기능 확장이 아니라, 6.x 사건 패키지 운영 이후 발생하는 보완 요청 흐름을 별도 로드맵으로 분리해 구현하는 단계다.

따라서 다음 작업은 코드 구현이 아니라 다음 문서 작성이다.

AI법친 7.1-B — 보완 요청 워크플로우 정의서
