## 0. 기입 원칙

- 이번 초안은 **전제 2 해소 선언용이 아니라 1차 기입용**이다.
- 상위 판정표 **§6-1**의 **DENY 유지·잠금** 판정을 **임의로 바꾸지 않는다.**
- 각 행은 UI / API / 서버 / 운영·감사 4축 중 **실제 근거가 확인된 것만 ☑**, 나머지는 **☐** 또는 **—**로 둔다.
- §6 DENY는 **문서상 금지 규칙이 이미 잠겨** 있으므로, 이번 초안의 핵심은 **「실제 제품 통제선에서 어디가 입증됐는가」**이다. 문서상 금지라는 이유만으로 4축에 **자동 ☑**를 주지 않는다.
- `check-status --scope case` 는 **휴리스틱 참고값**일 뿐 **직접 판정 근거로 쓰지 않는다.**

## 8. DENY-8 — `DELETED` → `CREATED`

**1차 기입안**

| UI | API | 서버 | 운영·감사 |
| --- | --- | --- | --- |
| ☐ | ☐ | ☐ | ☐ |

**실무 한 줄**

`DELETED` → `CREATED`는 현재 기준에서 **직접 허용 전이 아님**으로 읽고, **DENY-8**은 일반 재진입 금지·**OPEN-4**는 복원·재생성·삭제 후 처리 설명 축을 맡는 구조로 유지한다 ([227]). `CREATED`는 §6-1 **예시 토큰**이며, **soft delete·복원·운영 예외**와 교차할 수 있어 UI/API/서버/운영 **전 축 입증**은 후속이다.

**근거 메모**

- §6-1 및 **§11·운영**과 **교차 가능성**이 있어 **전 축 ☐**가 적절하다.
- soft delete·복원·감사 추적이 얽힐 수 있으므로 운영·감사도 **—가 아니라 ☐**로 둔다.
- [227]에서 **OPEN-4**와의 역할 분리(금지 축 vs 설명·추적 축)를 문서상 고정했다.
- **[228]:** `softDeleteCaseService`는 권한 후 `updateCaseById`→`DELETED`로 진입 확인. **상태별 삭제 제한**·`DELETED`→`CREATED` 직접 복구는 현재 파일 기준 미확인.

## 10. 이 초안의 해석 규칙

- 이 표는 **DENY 8행 전부 미해소 유지**(§6-1 역점검 고정)를 전제로 한다.
- §6-1에서 이미 **문서 잠금**은 강하게 걸렸으므로, 이번 **☐** 들은 「문서가 약하다」는 뜻이 아니라 **「실제 통제 연결 입증이 아직 비어 있다」**는 뜻이다.
- ☐가 많다고 바로 **구현 오류**로 단정하지 않는다.
- 다만 현재 시점에는 UI/API/서버/운영 **중 어느 축에서 금지 규칙이 실제로 보장되는지** 추가 확인이 필요하다.

## 11. 문서에 붙일 한 줄 결론

§6 DENY-1 ~ DENY-8 1차 기입은 **모두 미해소 유지** — 문서상 **금지 규칙은 잠겼지만**, 각 금지 행이 실제 제품 통제선(UI·API·서버·운영) **어디에서 차단·추적되는지**는 아직 **추가 입증**이 필요하다.
