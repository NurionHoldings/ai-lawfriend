# EXECUTION_SEQUENCE

## 목적

AI법친 프로젝트의 작업 순서를 흔들림 없이 유지하기 위해, **기준문서 잠금 → 구현 정렬 → 검증**의 순서를 고정한다.

이 문서는 다음 원칙을 전제로 한다.

- 상태 이름의 유일 기준은 `prisma/schema.prisma` 와 `src/lib/definitions/case-status.ts` 이다.
- 옛 메모, 패치 노트, 위키, 슬랙 기록은 단독 기준으로 사용할 수 없다.
- 새 기능 추가보다 **기준문서 잠금**이 우선이다.
- 구현은 정의서를 따르도록 재정렬한다.

---

## 1. 전체 실행 원칙

### 1-1. 선행 원칙

모든 작업은 아래 순서를 지킨다.

1. 기준 파일 존재 확인
2. 상태 기준 확인
3. 기준문서 작성/잠금
4. 구현과 기준문서 비교
5. 불일치 수정
6. 그 이후 신규 기능 검토

### 1-2. 금지 원칙

아래 행동은 금지한다.

- 상태 기준 파일 확인 없이 상태 문서 작성
- 옛 메모 용어를 현행 상태처럼 사용
- 기준문서 잠금 전에 기능부터 추가
- 정의서와 구현이 다른데도 “나중에 정리”로 미루기
- 본선 코드가 아닌 구 패치셋을 현행 기준으로 간주

---

## 2. 작업 시작 전 선행 게이트

### Step 0. Canonical Source 검증

작업 시작 전에 반드시 아래를 확인한다.

- `prisma/schema.prisma`
- `src/lib/definitions/case-status.ts`

둘 다 있어야만 상태 관련 작업을 진행할 수 있다.

### 실행 명령

```bash
npm run verify:canonical-sources
```

또는

```bash
npx tsx scripts/verify-canonical-sources.ts
```

다른 폴더를 검사할 때는:

```bash
npx tsx scripts/verify-canonical-sources.ts --root "C:\path\to\extracted"
```

### 판정 기준

- **성공(exit 0):** 다음 단계 진행 가능
- **실패(exit 1):** 상태 관련 문서화, 비교, 패치 작업 중지

## 3. 실행 순서(고정)

### Phase 1. 도메인 기준문서 잠금

**목적:** 개발팀이 같은 기준으로 해석하도록 핵심 정의를 먼저 확정한다.

**순서**

1. 상태값 정의서
2. 사건 라이프사이클 정의서
3. 권한정의서
4. 질문셋 정의서
5. 문서 템플릿 정의서

**산출물:** 위 정의서 각각

**완료 기준**

- 각 문서가 현행 코드 기준으로 작성됨
- 상태명/역할명이 본선 코드와 충돌하지 않음
- 구 용어 사용 시 매핑표가 붙어 있음

### Phase 2. AI 출력 기준 잠금

**목적:** AI가 생성하는 결과물의 허용 범위와 책임 경계를 명확히 한다.

**순서**

1. AI 출력 정책서
2. 고지문/면책문구 정의서
3. 사건 요약 출력 명세서

**산출물:** 위 정의서 각각

**완료 기준**

- AI 초안과 승인본이 구분되어 있음
- 사용자 노출 문구가 통일됨
- 민감정보/면책 기준이 명시됨

### Phase 3. 데이터 구조 문서화

**목적:** DB, API, 화면이 서로 다르게 해석되지 않도록 입력/출력과 구조를 고정한다.

**순서**

1. 입력/출력 데이터 정의서
2. 첨부자료 분류 기준서
3. DB 상세 설계 초안

**산출물:** 위 문서 각각

**완료 기준**

- 주요 API 입력/출력 구조가 문서화됨
- 첨부 분류 규칙이 명확함
- 핵심 테이블 관계가 설명됨

### Phase 4. 화면/API 기준 문서화

**목적:** MVP 주경로를 화면과 API 기준으로 다시 잠근다.

**순서**

1. 화면 우선순위표
2. API 명세 1차본

**산출물:** 위 산출물 각각

**완료 기준**

- MVP 주경로가 화면 순서로 고정됨
- API 계약 초안이 확정됨
- 화면과 API 명세가 상호 연결됨

### Phase 5. 정의서 대비 구현 역점검

**목적:** 현재 구현이 기준문서와 어긋난 부분을 찾아 우선순위대로 정리한다.

**점검 항목**

- 상태값 불일치
- 역할명 불일치
- 질문셋 구조 불일치
- 문서 템플릿 구조 불일치
- API 요청/응답 키 불일치
- 화면 문구/상태 라벨 불일치
- 승인/잠금/복원 정책 불일치

**산출물**

- 정의서 대비 구현 불일치 역점검표
- 수정 우선순위표
- 파일별 수정 대상 목록

**완료 기준**

- 불일치 항목이 누락 없이 표로 정리됨
- 수정 우선순위가 정해짐
- 바로 패치 가능한 상태가 됨

### Phase 6. 구현 재정렬 패치

**목적:** 현재 구현을 잠근 정의서 기준으로 정렬한다.

**우선순위**

1. role / permission 통합
2. case status / lifecycle 통합
3. question set / interview contract 통합
4. document template / paragraph policy 통합
5. API response contract 통합
6. 화면 문구/버튼/상태 라벨 통합

**산출물**

- 패치 지시서
- 파일별 diff 초안
- 검토용 체크리스트

**완료 기준**

- 정의서와 실제 구현의 핵심 충돌이 해소됨
- 상태/권한/전이 로직이 통일됨
- UI 문구와 API 계약이 기준문서와 맞아떨어짐

## 4. MVP 기준 주경로

아래 경로를 MVP의 기준 흐름으로 본다.

1. 변호사 가입/승인
2. 로그인
3. 대시보드
4. 의뢰인 초대/가입
5. 사건 생성
6. AI 인터뷰
7. 사건 요약
8. 문서 생성
9. 사건 상세
10. 보완 요청
11. 관리자 승인

**원칙**

- 기준문서는 위 흐름을 기준으로 작성한다.
- 구현 재정렬도 위 흐름을 기준으로 우선순위를 둔다.
- 운영 고도화 기능은 MVP 기준 경로와 충돌하지 않도록 후순위 정렬한다.

## 5. 중간 점검 순서

작업 중간마다 아래 순서로 점검한다.

### 점검 A. 기준 확인

- `verify:canonical-sources` 통과
- 상태 기준 파일 존재
- 구 상태명 혼용 여부 확인

### 점검 B. 문서 진행 상태 확인

- 현재 Phase가 어디까지 왔는지 확인
- 이전 Phase가 잠겼는지 확인
- 미완 문서와 다음 문서를 구분

### 점검 C. 구현 정합성 확인

- 정의서와 코드 상태명 일치
- 권한명/역할명 일치
- API/화면 라벨 일치

### 점검 D. 다음 세션 브리핑 생성

- 오늘 완료 항목 기록
- 다음 작업 1순위 기록
- 주의할 불일치 항목 기록

## 7. 옛 자료 사용 절차

옛 자료를 참고해야 할 경우 아래 절차를 따른다.

1. 현행 기준 파일 먼저 확인
2. 옛 자료의 상태명/역할명을 추출
3. 매핑표 작성
4. 현행 기준으로 치환
5. 치환 완료 후에만 패치/문서화 반영

**주의:** 매핑표 없이 옛 자료를 바로 반영하지 않는다.
